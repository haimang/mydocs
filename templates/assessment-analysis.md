# Assessment 模板簇 · analysis（单调查面深度评估）

> **模板簇**：`assessment`（**独立簇，完全自包含；不属于 eval 家族，无 eval 血缘**——本模板不继承任何 eval 共有脊、不声明 `eval/*` 文档性质、不从 eval 模板抽取模块；所有矩阵/图例/纪律均在本文内自定义）。
> **簇内成员**：[[assessment-index]]（阶段编排器）+ `assessment-analysis`（本文，单面深度评估）。本文的上游是 index 钉定的**某一个调查面** + **冻结分母**；下游喂规划/决策/执行。
>
> **一句话用途**：对**一个**调查面（子系统/能力面）做一次 measure-first 的深度评估——把现状测准、把"可借鉴的参考"过滤成"能落地的机制"、把缺口编号成台账、把净新契约与跨功能不变量草拟出来、把验收格栅与 owner-gate 候选标出来，**为下游设计/规划/执行供一份可核验的统一制品**。
> **何时用**：[[assessment-index]] 已冻结分母、本面已在评估地图登记，开始逐面深挖时；每个调查面产出一份。
> **何时不用**：① 还在编排"有哪些面、谁先谁后" → 用 [[assessment-index]]；② 已在排"怎么做/何序/怎么测"的执行任务 → 那是下游执行计划，不在本簇。
>
> **调查纪律（assessment 簇共有脊 · 两模板一致）**：
> 1. **定位**：measure-first 分析制品，为下游*设计 / 规划 / 决策 / 执行*供料；**不排执行序、不写落地验收、不替 owner 裁决**。
> 2. **measure-first 红线**：一切"现状"以 **HEAD 实测**为准（每条带 `path:line` + 可复现命令）；引用 index 的**冻结分母**，不另估。
> 3. **零决策、只 MARK**：需 owner 拍板者列为 owner-gate 候选并 MARK（§10.2），裁决交下游决策登记，**不在本文冻结**。
> 4. **借鉴 ≠ 机制**：任何"参考怎么做"必须经 §7 **substrate-fit / 技术路线过滤**，确认能落到本仓现实再算数；反例显式登记（§8.1）。
> 5. **防假绿**：缺口带证据编号（§4）；验收格栅草案（§9）显式封堵 fake-green；声称 vs 实测对不齐者点名（§11）。
>
> **图例（本簇自有，全部 inline 定义，勿与他处混用）**：
> - **借鉴 verdict**（§3 矩阵用）：`✅ 借`（可直接借）/ `🔶 部分借`（借思路不借机制）/ `⛔ 反例`（要避开）/ `🆕 净新`（无先例，须从零写）。
> - **复用判定**（我方代码处置）：`✅ 复用` / `♻️ 重 substrate`（在既有底座上重建）/ `🆕 净新`。
> - **缺口严重度**：`S1 阻断` / `S2 重要` / `S3 次要`。
> - **置信分层**：`HEAD 实测 > 仓内文档锚 > 外部参考`（高→低）。
> **状态词汇（本簇自有）**：`draft`（在写）→ `reviewed`（已复核）→ `frozen`（冻结分母 + 借鉴矩阵 + 契约草案锁定，作下游基线）→ `superseded`。
> **占位符约定**：`{UPPER_SNAKE}` backtick 包裹；`[核心]` 必填 / `[可选]` 视面删除。
> **流水线位置**：本文 = **站②**（上游 = 站① [[assessment-index]]）→ 下游 = `{PHASE}/proposed-planning.md` 等。
